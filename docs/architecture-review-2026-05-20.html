Where the two agents converged

On the Agent concept

R1 noticed cli_agent_orchestrator/agent.py (durable config) and runtime/agent.py (live state) compete, while models/ conspicuously lacks an agent.py. R2 noticed AgentRuntimeHandle leaks freshness, restart, inbox-rehome, and workspace-switch facts. These are the same architectural fault from two altitudes — synthesised as candidate #2.

On Provider duplication

R1 saw get_status() copy-pasted across 7 adapters. R2 saw JSON state persistence duplicated in Claude/Codex and transcript parsing duplicated in Q/Kiro. Three faces of the same shallow base — synthesised as candidate #3, where BaseProvider grows three private collaborators.

Candidate #1 · Deepening opportunity

Give the Inbox concept a deep home.

Strong R1 #1

Inbox is a domain noun in CODEBASE.md and docs/api.md, but inside src/ it splits into eight shallow modules across three packages — each thin, each with its own interface. The deletion test concentrates: collapse them, complexity doesn't reappear at callers.

Files

  • services/inbox_service.py · 176
  • clients/inbox_store.py · 605
  • provider_conversations/inbox_access.py · 430
  • …/inbox_authorization.py · 173
  • …/inbox_bridge.py · 493
  • …/inbox_read_presentation.py · 35
  • …/persistence.py · 557
  • …/reply_service.py · 445

~2,880 lines · 8 modules · 3 packages.

Problem. Each Inbox module is shallow — interface nearly as wide as implementation. inbox_read_presentation.py is 35 lines of formatting; inbox_authorization.py is a pass-through. Together they fragment one concept across three packages, forcing every caller to learn which package owns which concern.
Solution. Collapse into one inbox/ package exposing one deep Inbox interface: send · read · deliver_when_ready · authorize. Persistence, authorization, presentation, bridge, and watchdog become private collaborators — internal seams, not external ones. Pairs with candidate #6 (TerminalReadinessMonitor as a private collaborator).

Benefits

  • locality: delivery, auth, formatting bugs land in one place
  • leverage: workflow patterns (handoff, baton, future review loops) call one Inbox interface, not seven
  • the interface is the test surface: exercise end-to-end against real SQLite + fake provider, not 6 directories mocking 5 collaborators each

Candidate #2 · Deepening opportunity

Consolidate the Agent concept across model and runtime.

Strong R1 #3 + R2 #1

Per your stated mental model (agent → instance, ≤1 live terminal): the code carries the bones but not the depth. Both agents flagged this — from different layers. Tackle in two steps: first move the model, then deepen the runtime.

Files

  • cli_agent_orchestrator/agent.py · 1,257 (durable config)
  • runtime/agent.py · 1,397 (handle + freshness)
  • services/terminal_service.py
  • services/inbox_service.py
  • models/ · has terminal, session, inbox, flow — no agent.py
Problem (Step A — R1). Top-level agent.py owns FrozenConfigDict, AgentConfigError, AGENTS_ROOT, validation. runtime/agent.py imports from it. Meanwhile models/ already holds Terminal, Session, Inbox, Flow — Agent is the conspicuous absentee. Deletion test: erasing top-level agent.py would force its contents into models/agent.py, alongside its peers.
Problem (Step B — R2). AgentRuntimeHandle leaks freshness, restart, inbox-rehome, workspace-switch, and state-file ordering across its seam — tests cross private boundaries and inspect raw DB rows.
Solution. (A) Move durable-config code from cli_agent_orchestrator/agent.py into models/agent.py. (B) Deepen runtime/agent.py so AgentRuntimeHandle stays small and the implementation absorbs restart, rehome, freshness, event facts.

Benefits

  • locality: "what is an agent" lives in one model file; restart/rehome rules concentrate in one runtime
  • leverage: one seam for callers and tests; future workflow patterns compose on a clean handle
  • AI-navigability: an agent reading the repo finds models/agent.py adjacent to its peers, not at the package root

After — one model, one deep runtime

Candidate #3 · Deepening opportunity

Deepen BaseProvider: three private collaborators absorb the duplication.

Strong R1 #2 + R2 #3 + R2 #5

Three separate findings, one root: providers/base.py declares abstracts but owns no behaviour, so every adapter re-implements control flow. R1 flagged get_status() copy-pasted in 7 files; R2 flagged JSON state persistence duplicated in Claude/Codex, and ANSI/prompt/permission parsing duplicated in Q/Kiro.

Files

  • providers/base.py · 373
  • codex.py · 995
  • claude_code.py · 873
  • gemini_cli.py · 792
  • kimi_cli.py · 651
  • copilot_cli.py · 472
  • kiro_cli.py · 449
  • q_cli.py

7 adapters duplicating control flow.

Problem. BaseProvider is shallow — its interface declares get_status, get_idle_pattern_for_log, extract_last_message_from_script as abstract, but the implementation of each ships inside every adapter. A bug in tail-capture or escape-sequence handling lives in 7 files (inbox_service already warns the pattern is "insufficient"). Persistence has the same shape — two adapters, two copies. Transcript parsing — two adapters, two copies.
Solution. Deepen BaseProvider by introducing three internal seams, each with one deep collaborator:
  1. PromptStateMachine — consumes a ProviderPromptSpec per adapter; owns tail / scan / classify
  2. RuntimeStateStore — owns JSON persistence, schema envelope, load/save/clear
  3. TranscriptParser — owns ANSI strip, prompt positions, permission detection, response slice
Adapters declare data (regexes, escape rules, transcript shape) — not control flow.

Benefits

  • locality: escape-handling, schema, parser bugs each fix in one place
  • leverage: 7 adapters share one deep implementation; new adapter = a spec, not a loop
  • two adapters make a real seam: the seam exists today but is being re-derived inside each — make it carry weight

Candidate #4 · Deepening opportunity

api/main.py is a 2,860-line shop with 60 routes.

Strong R1 #4

Sessions, terminals, agents, workspaces, workspace-teams, batons, monitoring, providers, and tool descriptors all share one file. Decorators, request models, error handlers, and dependency wiring mix freely. Tests in test/api/ already split by domain — the seam exists in your head, not in the directory.

Files

  • api/main.py · 2,860 lines · 60 routes

Routes per domain

  • workspaces · 7
  • workspace-teams · 8
  • agents · 9
  • sessions / terminals · 12
  • batons · 8 (feature-gated inline)
  • monitoring · 6
  • providers / tools · 10
Problem. Touching one domain forces a scroll past the others. Feature-flagged routes (baton) get inline if CAO_BATON_ENABLED guards. The deletion test fails clearly here: deleting the file concentrates nothing — it just relocates 60 routes, which is the whole point.
Solution. Split into FastAPI APIRouters by domain noun: api/routers/{sessions,terminals,agents,workspaces,workspace_teams,batons,monitoring,providers,tools}.py. main.py shrinks to app construction, middleware, exception handlers, router inclusion. Each router file owns its request models, error mapping, dependency wiring. Pairs with candidate #7 — the whole baton router gets included or skipped at app build, no inline flag checks.

Benefits

  • locality: a baton-routing bug lives entirely in routers/batons.py
  • leverage: each router gets exercised against a real TestClient in its own file — matches what the tests want to be
  • cleaner feature gating: wire-time, not call-time

Candidate #5 · Deepening opportunity

Collapse terminal launch choreography into one deep module.

Strong R2 #2

terminal_service.create_terminal_for_agent advertises a generic create-path, but the real flow requires an agent launch and exposes 7 ordered steps to tests. The cleanup ordering between tmux, DB, and provider leaks across the seam.

Files

  • services/terminal_service.py · 665
  • providers/manager.py
  • clients/tmux.py
  • test/services/test_terminal_service_*.py
Problem. Seven thin bands — generate ids, resolve agent material, prepare provider runtime, create tmux session, write terminal row, initialize provider, pipe log file — each visible to tests via patching. The interface is wide because every band is exposed; the implementation is thin because each band does one thing. Shallow on both ends.
Solution. Deepen launch around the agent-terminal workflow. launch_agent_terminal becomes one deep module owning launch order, rollback, logging, and provider preparation. tmux / DB / provider become internal seams (private adapters), not points tests patch through.

Benefits

  • locality: cleanup rules live once — no more "did I roll back the tmux session if DB write fails?"
  • leverage: fewer launch patches in tests; the seam matches the implementation
  • generic create-path either becomes real or disappears

Candidate #6 · Deepening opportunity

Inbox watchdog and baton watchdog solve the same problem twice.

Worth exploring R1 #5

Both watchers ask: "is this terminal ready to receive something now?" Inbox uses watchdog.Observer on the log file; baton uses an async scan loop. Same concern, two mechanisms, two failure surfaces. Best landed after Inbox (#1) — as a private collaborator of the deep Inbox module.

Files

  • services/inbox_service.py:39 · watchdog Observer
  • services/baton_watchdog_service.py:312 · async scan loop
  • services/baton_service.py
Problem. Both consume terminal readiness, but neither names that concept. Adding a new TUI provider means changing both. Tests mock both layers separately.
Solution. Introduce a TerminalReadinessMonitor seam: subscribe(terminal_id, callback). Inbox and baton register callbacks; one deep implementation owns "how do I know this terminal is idle" (file-watching, polling, or a future push-based provider hook).

Benefits

  • two adapters justify the seam: Inbox + baton both consume it — real, not hypothetical
  • leverage: future push-based provider hooks plug in here
  • locality: "watcher missed an event" has one place to debug

Candidate #7 · Deepening opportunity

Feature flags are checked inline at every site.

Worth exploring R1 #6

CAO_BATON_ENABLED and CAO_ENABLE_WORKING_DIRECTORY are checked across HTTP routes, MCP server registration, CLI command attachment, and watchdog startup. The next experimental feature will repeat the pattern — and your stated preference is enforcement via automation, not multi-site discipline.

Files

  • services/baton_feature.py
  • api/main.py:2394
  • mcp_server/server.py:72
  • cli/commands/agent.py
  • services/baton_watchdog_service.py
Problem. 4 sites for baton, 2 for working-directory. Graduating baton means deleting 4 inline checks in coordination — a multi-site discipline.
Solution. One FeatureRegistry. Routers, MCP tools, CLI commands, watchdogs each declare a feature requirement at registration. The registry decides once at startup whether to wire them. Pairs naturally with candidate #4.

Benefits

  • locality: graduating a feature = removing one registry entry
  • leverage: next experimental feature drops into the same shape
  • tests flip the registry, not 4 env-var reads

Candidate #8 · Deepening opportunity

Deepen the dashboard HTTP-shape adapter.

Worth exploring R2 #4

web/src/api.ts hand-copies large server shapes, endpoint strings, encoding rules, and timeouts. Both the TS adapter and the Pydantic route are shallow — interface nearly matches implementation because both encode the same shape twice.

Files

  • web/src/api.ts
  • api/main.py
  • web/src/test/api.test.ts
Problem. Shape lives twice — once as Pydantic, once as TypeScript. Every server-side change is a multi-site discipline. Tests verify shape, not behaviour.

Benefits

  • leverage: fewer brittle mocks

Top recommendation

Start with #1 — Give Inbox a deep home.

Both source agents converged on the Agent concept as a top concern, but Inbox sprawl is the higher-leverage move first. The workflow construction phase (per your memory) will compose every pattern — handoff, baton, future review loops — on Inbox delivery. Building those patterns on eight shallow Inbox modules will spread workflow logic across the same three packages.

  • Collapses the largest concept-sprawl in the codebase (~2,880 lines · 8 modules · 3 packages).
  • Unlocks #6 cleanly — TerminalReadinessMonitor becomes a private collaborator of Inbox, not a sibling.
  • Strips noise from tool_service.py and mcp_server/server.py, making #4 simpler.
  • Matches the agent → instance model: each instance has an Inbox, not eight files implementing parts of one.

After Inbox lands: #2 (Agent consolidation) and #3 (Provider depth) are independent and can run in parallel. #4 and #5 become natural follow-ups once Inbox has shown what a deep module looks like in this codebase.
